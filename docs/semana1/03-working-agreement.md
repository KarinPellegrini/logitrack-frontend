# Working Agreement — LogiTrack

## 1. Organización del trabajo del equipo

**Contexto**  
El equipo LogiTrack está conformado por tres integrantes que deben coordinarse para la realización de las tareas y desarrollo del MVP. Para evitar confusiones, tareas duplicadas o falta de seguimiento del trabajo, se vio necesario establecer un medio para gestionar el trabajo del equipo.

**Decisión**  
El equipo utilizará un tablero Kanban para organizar las tareas del proyecto. Cada tarea tendrá asignado un responsable y una breve descripción.

**Alternativas consideradas**
- Gestionar las tareas únicamente por Discord y WhatsApp.
- Utilizar documentos compartidos sin el uso de un tablero visual.

**Consecuencias**
- Mayor visibilidad del avance del proyecto.
- Reducción del riesgo de tareas duplicadas.
- Mejora en la organización y seguimiento del trabajo del equipo.

---

## 2. Comunicación del equipo

**Contexto**  
El equipo LogiTrack necesita un canal de comunicación para resolver sus dudas y coordinar el avance diario del proyecto.

**Decisión**  
El equipo utilizará Discord y WhatsApp como canales principales de comunicación: WhatsApp para el día a día y coordinación, y Discord para el envío de información de forma más ordenada y para realizar reuniones internas.

**Alternativas consideradas**
- Utilizar únicamente reuniones presenciales.
- Utilizar plataformas como Zoom o Google Meet.
- Coordinar comunicación exclusivamente por correo electrónico.

**Consecuencias**
- Comunicación rápida y directa entre los integrantes.
- Comunicación ordenada.
- Mayor visibilidad del progreso diario.

---

## 3. Gestión del repositorio y control de código

**Contexto**  
El equipo de LogiTrack requiere la utilización de un repositorio de código compartido para almacenar el trabajo del equipo y permitir la colaboración entre los integrantes de una forma adecuada.

**Decisión**  
El equipo utilizará GitHub como repositorio principal del proyecto.

**Alternativas consideradas**
- Compartir archivos manualmente entre los integrantes.
- Utilizar almacenamiento en la nube sin control de versiones.
- Utilizar otras plataformas como GitLab o Bitbucket.

**Consecuencias**
- Historial claro de cambios en el proyecto.
- Mejor control de versiones del código.
- Requiere conocimientos de Git por parte del equipo.

---

## 4. Toma de decisiones técnicas

**Contexto**  
Al momento de surgir desacuerdos técnicos durante el desarrollo del proyecto entre los integrantes sobre herramientas, diseño o implementación.

**Decisión**  
Si no se logra consenso en una decisión técnica, el equipo realizará una votación simple entre los integrantes al inicio de la reunión semanal o por medio del grupo oficial de WhatsApp si no hay una reunión cercana o es una decisión que no puede esperar.

**Alternativas consideradas**
- Dejar las decisiones técnicas a criterio de un único integrante.
- Votar la decisión fuera de llamada, privando de poder argumentar el porqué de la decisión.
- Delegar todas las decisiones al Product Owner.

**Consecuencias**
- Las decisiones técnicas son debatidas y elegidas en consenso por todos los integrantes del equipo.
- Se reduce el riesgo de discusiones prolongadas.
- El equipo mantiene un proceso de decisión transparente.

---

## 5. Gestión de bloqueos y redistribución de tareas

**Contexto**  
Los integrantes del equipo tienen diferentes responsabilidades académicas, laborales y personales, lo que podría generar bloqueos o retrasos en algunas tareas.

**Decisión**  
Si un integrante detecta que no podrá completar una tarea asignada, debe informar al equipo lo antes posible para redistribuir el trabajo. El equipo priorizará siempre las tareas críticas del MVP.

**Alternativas consideradas**
- Mantener las tareas asignadas ignorando los inconvenientes personales.
- Ignorar bloqueos hasta el último día de entrega.

**Consecuencias**
- Mayor flexibilidad en la gestión del trabajo.
- Reduce el riesgo de retrasos en el proyecto.
- Requiere buena comunicación y confianza entre los integrantes.
